# 1️⃣ “Pool may not be initialized or parameters are invalid”

This is from **Add Liquidity**.

### What this means technically

In Uniswap v4, a pool must:

1. Be created (via PoolManager)
2. Be initialized with a starting price
3. Have correct pool key parameters:

   * currency0
   * currency1
   * fee
   * tickSpacing
   * hooks address

If any of those are wrong OR the pool was never initialized with `initialize()`, liquidity mint will revert.

### Most likely causes in your case

* Pool exists in UI but not actually created onchain.
* Token ordering mismatch (currency0/currency1 reversed from what PoolManager expects).
* Wrong tickSpacing for that fee tier.
* Using a fee tier that does not match the pool you think exists.
* Trying to add liquidity before initialize() was ever called.

This is not a UI bug — it’s a **pool lifecycle issue**.

# 3️⃣ Swap Simulation Failed

> Unable to decode signature 0x486aa307

This is the big one.

This means:

You are calling a function that does NOT match the ABI of the contract you think you’re calling.

That 4-byte signature:

```
0x486aa307
```

does not exist in the ABI you provided to viem.

That happens when:

* You are calling PoolManager.swap with wrong ABI
* You are using Router ABI against PoolManager address
* You are encoding swap params incorrectly
* The contract at that address is not the contract you think it is

This is a **wrong contract interface or wrong address issue.**

Not gas.
Not slippage.
Not liquidity.

ABI mismatch.
